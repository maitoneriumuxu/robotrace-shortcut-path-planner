## 3. AALPの単位確認

### 3.1 元コードの計算

`Angular_Acceleration_Limit_Planner()`は探索ログの`yawAccel_seg`を`alpha_meas`とし、次を計算する。

```text
v_allow = SEARCH_RAN_SPEED × sqrt(max_AALP / abs(yawAccel_seg))
```

その後、最低・最高速度へクランプし、`set_speed_min()`でGFCPとAALPの小さい方を採用する。

### 3.2 数値単位

`mpu_calculation.c`は1 ms周期で次を実行する。

```c
MST.yawAccel = MST.gyro_z - yaw_rate0;
```

`gyro_z`はdeg/sだが、差分を0.001 sで割っていない。`yawAccel_10ms_avg`を距離セグメントへ保存し、DUMPヘッダでは`yawAccel[deg/s2]`と表示する。このためヘッダ上の単位と実際の数値生成は一致しない。

本PC版では`max_AALP=100`を勝手にrad/s²やdeg/s²へ読み替えず、「deg/sの1 ms当たり変化量」という元コードの数値単位を再現した。

### 3.3 新経路への変換

探索速度3.6 m/s一定、したがって探索時縦加速度`a=0`として、幾何式を使う。

```text
alpha_rad_s2 = a × curvature + v_ref² × dcurvature/ds
             = 3.6² × dcurvature/ds

alpha_firmware = abs(alpha_rad_s2) × 180 / π × 0.001
v_AALP = 3.6 × sqrt(100 / alpha_firmware)
```

最後に3.6～13.0 m/sへクランプし、GFCPとのminを取る。今回の4経路ではAALPが直接支配した点は0点で、GFCP単独とAALP互換の予測差は0.17 ms未満だった。主結果はAALP互換込み、GFCP単独は参考値として保持する。

## 4. ATTACK速度モデル

| 設定 | 値 |
|---|---:|
| `R10_speed` | 3.6 m/s（ユーザー指定） |
| 最低速度 | 3.6 m/s |
| 始端・終端速度 | 3.6 m/s |
| 最高速度 | 13.0 m/s |
| `speed_root` | 0.33 |
| 最小加速度 | 20.0 m/s² |
| 最大加速度 | 55.0 m/s² |
| 減速度 | 55.0 m/s² |
| `break_kp` | 1.00 |
| `max_AALP` | 100（§3の数値単位） |
| 最大加速度側omega閾値 | 300 deg/s |
| 最小加速度側omega閾値 | 1500 deg/s |
| `jerk` | 3.0（Dutyモデルがないため未使用） |

GFCP:

```text
v_GFCP = 3.6 × abs(radius_mm / 100.0) ^ 0.33
```

omega依存加速度:

- `|omega| <= 300 deg/s`: 55 m/s²
- `|omega| >= 1500 deg/s`: 20 m/s²
- 中間: 55から20 m/s²へ線形補間

`omega=v×curvature`から各点の許容加速度を作り、実点間距離による前向き加速スキャン、55 m/s²による後向き減速スキャンを固定4回行う。改善後経路の計算上の最大`|omega|`は約2418 deg/sであり、1500 deg/sを速度ハード制約にはしていない。

## 6. 改善後の経路生成

### 6.1 初期値

- 原ラインのゼロオフセット
- Elastic Band XYを原ライン同一index法線へ射影したオフセット
- 基準表示用に再現した旧時間選択型#7オフセット

Elastic Band自身と旧#7も最終候補として別に保持する。改善候補が遅い、または最終幾何検査に違反する場合は、予測時間が最短の有効フォールバックを採用する。

### 6.2 自動検出

原コースの固定幅曲率から、最大曲率の85%以上にある局所最大を検出する。2.5 m以内に3点以上あり、隣接ピークの曲率符号が交互に変わるグループを優先する。この入力ではindex`1498, 1590, 1701`が選ばれたが、コードにこの番号は書かれていない。

### 6.3 長窓候補

各検出中心へ端点0のraised cosineを置く。変更するのは原ライン同一indexの法線オフセットだけである。

粗探索の固定6候補:

- 200 mm / ±5 mm
- 500 mm / ±10 mm
- 800 mm / ±20 mm

各3初期値について「変更なし+6候補」を1本ずつ評価し、最良記述子だけを保持する。その後、最良粗候補へ200 mm / ±2.5 mm、500/800 mm / ±5 mmの固定6候補を試す。合計27候補である。

採用記述子:

```text
旧#7初期 + 800 mm/+20 mm + 800 mm/+5 mm微調整
```

## 7. 幾何制約

改善後経路では最小半径100 mmを不合格条件にしていない。50/60/70/80/100 mmの新しい半径ハード制約も置いていないため、半径感度で採否を変える比較は不要だった。

使用する制約:

- 横オフセット75 mm
- 最大点間距離20 mm
- 1点オフセット変化10 mm
- 非有限値禁止
- 対応する原コース線分に対する逆行禁止
- 原コースより自己交差数を増やさない
- 最大`|dκ/ds|` 180 1/m²
- 始端・終端の元ライン復帰
- Frenet同一index再構成一致

180 1/m²は性能最適化値ではなく、旧#7の実測82.6 1/m²の約2.2倍に置いた数値異常ガードである。旧#7生成器だけは基準形状再現のため100 mm半径修復を残すが、長窓更新後には適用しない。

## 10. 計算時間とC移植観点

同一PCの代表値:

- Elastic Band生成: 約26 ms（420反復）
- 旧#7生成: 約58 ms（700反復）
- 長窓候補探索: 約0.49秒
- 速度計画: 1経路あたり主結果約18 ms、GFCP単独参考値込み約35～36 ms
- 候補評価: 27本
- 1候補: raised cosine適用1回、速度前後スキャン4反復
- 全処理の固定O(N)走査概算: 約3394回

旧実装の8候補評価+再生成約0.48秒に対し、長窓探索も約0.49秒で同程度に収まった。自己交差、曲率変化上限、Frenet一致などの全幾何検査は最終候補へ行い、探索中は局所制約だけを確認する。最終候補が不合格なら有効なElastic/旧#7へ戻る。

RX651で重い処理:

- 旧#7初期値を毎回700反復で作る部分
- 27候補それぞれの曲率、`powf`、`sqrtf`、4回前後速度スキャン
- PC検証用の自己交差空間格子

raised cosineは固定窓テーブル化できる。自己交差検査はPython幾何評価側に分離しており、C側の経路生成・速度スキャンはfloat32、固定幅、再帰なし、候補1本分、O(N)走査へ置換可能である。
